# 心语最终答辩演示顺序

本文档对应 `IMPLEMENTATION_PLAN.md` 步骤 `14.3`，用于把当前 MVP 收敛成一条可在答辩现场稳定展示的单链路。它不是 `13.3` 的全量人工走查替代品，而是更短、更适合现场直接执行的最终 runbook。

## 1. 目标

在 6-8 分钟内完成以下五个展示点：

1. 学生登录
2. 快速筛查局部展示
3. 高风险树洞提交
4. 后台队列复核
5. 模拟通知日志展示

这条链路优先证明三件事：

- 学生端主流程可进入并可操作；
- 高风险内容会被前台拦截，且绝不公开进入广场；
- 后台能对同一条案例完成人工复核、模拟通知与审计化处置。

## 2. 推荐环境

答辩现场默认使用下面这组配置，优先稳定性而不是“完全真实外网”：

```env
ENABLE_DEMO_LOGIN=true
ENABLE_MOCK_AI=true
SHOW_SEEDED_CASES=true
```

推荐原因：

- `ENABLE_DEMO_LOGIN=true`：学生端登录存在兜底入口；
- `ENABLE_MOCK_AI=true`：高风险树洞不依赖 DeepSeek 外网；
- `SHOW_SEEDED_CASES=true`：后台任意时刻都有可展示案例，不怕现场空列表。

推荐在正式开始前执行：

```bash
cd /Users/xingheluqi/Xin-Yu-College-Student-Mental-Health-Platform/backend
.venv/bin/python -m src.utils.seed_demo_dataset
```

## 3. 演示顺序

| 步骤 | 页面 / 系统 | 建议时长 | 现场目标 |
| :--- | :--- | :--- | :--- |
| 1 | `S02` 学生登录 | 40-60 秒 | 证明学生会话能创建 |
| 2 | `S05` 快速筛查局部展示 | 30-45 秒 | 证明问卷 UI、进度与低压设计 |
| 3 | `S12 -> S14` 高风险树洞提交 | 60-90 秒 | 证明高风险内容被拦截 |
| 4 | `A01 -> A03 -> A04` 后台复核 | 2-3 分钟 | 证明后台人工复核闭环 |
| 5 | `A04` 模拟通知日志 / 可选 `A07` | 45-60 秒 | 证明通知与审计可追溯 |

## 4. 主链路脚本

### 4.1 学生登录

推荐主路径：

1. 打开小程序，让系统经过 `S01` 自动跳到 `S02`。
2. 使用普通登录表单而不是演示账号，以便现场生成一条“新案例”：
   - 手机号：`13800001002`
   - 学院：`计算机学院`
   - 班级：`2026级1班`
3. 完成授权后进入 `S04 首页`。

推荐现场说明：

- “这里先证明学生端会话和授权链路可以正常进入。”
- “登录页会直接读取后端运行时特性，所以演示入口不是写死在前端里的。”

若主路径失败：

- 立即切换为 `使用演示账号进入`；
- 不再解释登录细节，直接进入首页继续后续链路。

### 4.2 快速筛查局部展示

1. 从 `S04` 进入 `S05 快速筛查页`。
2. 只展示前 1-2 题，不需要完整提交。
3. 重点展示：
   - 题卡布局
   - 进度结构
   - 风格克制、非游戏化
4. 返回首页。

推荐现场说明：

- “答辩现场不完整跑 70 题，这里只展示问卷体验和固定链路入口。”
- “完整解锁链路已经在前面的自动化测试和人工演练脚本中单独验证过。”

### 4.3 高风险树洞提交

1. 从 `S04` 进入 `S11`，再进入 `S12`。
2. 输入固定文案：

> 我真的不想活了，感觉没有任何意义。

3. 点击提交，确认进入 `S14 安全拦截页`。
4. 重点展示：
   - 没有“发布成功”
   - 有热线电话
   - 内容不会进入公开广场

推荐现场说明：

- “此处当前走的是本地 mock AI 容错路径，但状态机和后续工单闭环与真实外网模式一致。”
- “答辩重点不是大模型供应商是否在线，而是平台能否稳定把高风险内容拦截住。”

### 4.4 后台复核

1. 切到后台，使用 `platform.admin / Admin#2026` 登录。
2. 进入 `A02` 后直接点击“进入 A03”。
3. 在 `A03` 中打开最新的 `pending_review` 高风险树洞案例。
4. 在 `A04` 中依次展示：
   - 脱敏身份
   - 默认脱敏正文
   - 显式展开完整原文
   - AI 风险等级和触发短语
5. 填写并提交“确认高风险”：
   - 复核说明：`人工复核确认存在持续性危险表达。`
   - 干预说明：`已写入给辅导员的模拟联系说明。`

推荐现场说明：

- “后台敏感信息默认不直接展开，必须显式操作。”
- “从前台拦截到后台复核，这里形成的是同一条案例闭环，而不是两套互不相干的演示页面。”

### 4.5 模拟通知日志展示

1. 在 `A04` 确认成功提示已经出现。
2. 直接展示详情中的模拟通知日志区域。
3. 若时间允许，再进入 `A07`，筛选 `SYSTEM_CREATE_SIMULATED_NOTICE_LOG` 或对应案例进行补充展示。

推荐现场说明：

- “当前只写模拟通知日志，不触发真实外发，符合校园演示环境的边界。”
- “这一步证明的是处置记录可追溯，而不是现场真的给老师发消息。”

## 5. 兜底切换

若现场出现异常，按下面顺序降级，不要临场 improvisation：

1. 登录失败：切到演示账号登录。
2. 新案例未生成：继续使用 `SHOW_SEEDED_CASES=true` 下的预置高风险案例，从 `A02/A03` 直接进入后台复核段。
3. 学生端页面卡顿：直接保留 `S14` 截图或后台工单结果，继续讲后台闭环。
4. 后台筛选过多：先用 `pending_review` 过滤，再按最新案例打开。

## 6. 不建议现场做的事

- 不建议现场完整跑完 `SCREEN + SDS + SAS + SLEEP`。
- 不建议现场关闭 `ENABLE_MOCK_AI` 再赌外网状态。
- 不建议同时演示低风险树洞和高风险树洞两条学生链路，会稀释主线。
- 不建议在后台展开过多无关页面，避免把“单链路闭环”打散。

## 7. 完成标准

只有当下面五步能按顺序自然串起来时，`14.3` 才算真正可用：

1. 学生能进入会话并进入首页。
2. 问卷入口能被局部展示。
3. 高风险树洞能被稳定拦截到 `S14`。
4. 后台能找到同一条案例并确认高风险。
5. 模拟通知日志能被当场展示。
