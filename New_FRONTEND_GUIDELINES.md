# 心语新版前端指南

> 文档状态：当前 UI/UX 重绘候选方案，待用户确认后执行。
>
> 旧版 UI/UX 设计已保留在 `FRONTEND_GUIDELINES.md`，该文件不删除旧版设计，只定义下一阶段拟落地的新视觉与交互方案。

## 1. 文档目的

本文档用于定义心语学生端下一阶段的新版 UI / UX 方向。它不是对现有 `FRONTEND_GUIDELINES.md` 的直接覆盖，而是面向未来重构阶段的候选设计规范，重点回答：

- 如何把当前“单页向下滚动的入口式结构”升级为更接近主流 App 的导航结构；
- 如何在不引入即时通讯业务模型的前提下，借鉴 Telegram 与新一代 iOS 的底部悬浮导航语言；
- 如何在吸收微信官方小程序基础样式思想的前提下，避免把 UI 做成“脱离微信容器语境的外来 App 壳”；
- 如何在保持“平静、可信、低压”的心理健康产品气质下，完成更现代、更高完成度的学生端体验。

本文件当前主要服务于学生端小程序的新版视觉与交互设计；管理后台仍以现有 `FRONTEND_GUIDELINES.md` 为准。

参考来源：

- 当前新版 UI 设计除了参考 Telegram 风格的底部悬浮导航语言外，也明确参考微信官方设计团队开源的小程序基础样式库 `weui-wxss`：
  - https://github.com/Tencent/weui-wxss/
- `weui-wxss` 本身提供的是“与微信原生视觉体验一致的基础样式库”，核心价值在于：按钮、表单、列表单元、对话框、进度、提示、文章内容等基础界面元素都遵循微信用户已熟悉的阅读与操作节奏；
- 后续所有学生端新版视觉与交互设计，应理解为“Telegram / 新 iOS 导航语言 + 微信小程序原生一致性约束”的结合方案，而不是单纯照搬外部 App 的视觉外观。

特别说明：

- 本文件中当前仍保留的青绿色基调，只是因为现阶段尚未形成更成熟、更具辨识度的新品牌配色方案，因此暂时沿用项目既有的视觉记忆；
- 这套青绿色当前应被视为“过渡色系”，用于支撑新版导航结构、信息架构与交互层级的重构；
- 后续一旦形成更完整的新版主色系统，本文件中的品牌主色、导航高亮色与双主题色板都应继续修改，而不应被视为最终定稿。

## 2. 适用范围

- 学生端：微信小程序新版 UI / UX 重构
- 管理端：本文件不作为直接约束

## 2.1 当前 UI/UX 重绘实施方案（待确认）

### 2.1.1 方案结论

当前重绘不引入 React、Next.js、Tailwind CSS 或 `@mui/material`，也不改变微信原生小程序技术路线。Material UI / Material You 只作为视觉方法参考，实际实现继续使用：

- 微信原生小程序 `WXML / WXSS / JavaScript`
- 项目内自定义业务组件
- `miniprogram/styles/` 中的公共样式与设计令牌
- `admin` 端 Streamlit 原生组件与局部 CSS 注入

原因：

- `TECH_STACK.md` 已明确学生端为微信原生小程序，且明确不采用 React、Next.js、Tailwind CSS。
- `CODEX.md` 已明确本项目不使用 React，前端可复用 UI 必须走小程序原生组件化。
- Material UI 本质上是 React Web 组件库，直接引入会推翻当前小程序路线，收益低于风险。
- 当前问题主要是视觉模板感，不是业务状态机或技术栈不足，应通过设计系统和页面层级重绘解决。

### 2.1.2 设计方向

本阶段采用“Material You 灵感 + 微信小程序原生秩序 + 校园心理健康场景”的混合方向，内部命名为：

`静海校园 · Calm Material`

设计目标：

- 摆脱“通用白卡片堆叠”的模板感。
- 保留心理健康产品所需的平静、可信、低压和非诊断表达。
- 让四个一级频道具备各自的视觉记忆点。
- 保持微信小程序的阅读节奏、点击尺寸和基础控件习惯。
- 不把界面做成纯 Material 蓝白模板，也不做 Telegram / iOS 的像素级仿制。

视觉关键词：

- 柔和层级
- 雾面背景
- 校园晨光
- 圆润但不幼稚
- 内容流优先
- 风险提示严肃清晰

### 2.1.3 色彩与主题策略

旧版青绿色不再作为唯一品牌答案，而是退为过渡资产。新版拟采用更有辨识度的“静海蓝绿 + 晨光暖色 + 克制语义色”。

拟实施色彩 token：

| `calm-bg-mid` | `#EAF2F0` | 主背景中段 |
| `calm-bg-bottom` | `#F8F4EC` | 页面底部暖色过渡 |

| `calm-primary-soft` | `#DCEDEA` | 主色弱背景 |
| `calm-accent` | `#5C8FD6` | 当前导航、高亮信息 |
| `campus-warm` | `#E7A95D` | 进度、轻量引导、温暖点缀 |
| `ink-strong` | `#172B2E` | 一级标题 |
| `ink-body` | `#3E5558` | 正文 |
| `ink-muted` | `#7C8D8F` | 辅助文字 |
| `surface-glass` | `rgba(255, 255, 255, 0.82)` | 雾面卡片 |

本阶段先做明亮主题，不正式实现暗色主题。样式命名仍预留主题 token，避免后续二次重构。

### 2.1.4 信息架构策略

当前已形成 `测评 / 报告 / 树洞 / 我的` 四个一级频道，本阶段保留该结构，不改后端契约。

频道定位如下：

| 频道 | 新版定位 | UI 重点 |
| :--- | :--- | :--- |
| 测评 | 主线任务中心 | 70 题进度、必做 / 可选任务、下一步行动 |
| 报告 | 状态仪表盘 | 解锁状态、摘要结果、历史记录、下一步建议 |
| 树洞 | 校园匿名内容流 | 内容流、发帖主操作、支持反馈、授权禁用态 |
| 我的 | 个人与支持中心 | 身份、授权、帮助、演示环境、退出登录 |

不新增评论、私信、联系人、草稿、多媒体上传等社交功能。

### 2.1.5 组件重绘范围

本阶段优先重绘已有组件，不优先新增大量平行组件。

必须重绘：

- `app-shell`：升级背景氛围、底部安全区、页面雾面层级。
- `custom-tab-bar` / `floating-tab-bar`：升级为更完整的悬浮胶囊导航，补足图标感、选中态、按压态。
- `channel-header`：从说明型标题改为频道状态头部。
- `content-section-card`：从通用白卡改为分层卡片系统，支持 hero、glass、solid、warm、risk 等 tone。
- `primary-button / secondary-button / danger-button`：统一按钮高度、圆角、阴影、禁用态和按压态。
- `progress-bar`：改为更具 Material You 感的柔和进度轨道。
- `result-badge`：风险状态必须文字清晰，不能只靠颜色。
- `treehole-post-card`：从普通卡片改成内容流卡片。
- `question-card`：降低表单系统感，强化任务流程感。
- `confirm-modal`：保留原生克制感，减少装饰。

可局部增强：

- `quick-action-row`
- `safety-banner`
- `reaction-chip`
- `hero-card`

暂不新增复杂动画组件，不做手势拖拽导航。

### 2.1.6 页面实施方案

#### A. 测评频道首页

目标：从“问卷入口列表”升级为“70 题主线任务中心”。

具体实现：

- 顶部使用大号状态卡显示当前完成度和下一步。
- 必做问卷使用任务卡样式，展示完成态、剩余题量、风险摘要入口。
- UPI 使用弱化的可选辅助卡，不与必做任务同权重。
- 快捷入口保留，但压缩说明文字，避免首屏过度说明。

不改：

- 问卷顺序。
- 70 题解锁规则。
- UPI 可选规则。
- 页面跳转和提交接口。

#### B. 报告频道首页

目标：从“报告说明页”升级为“报告状态仪表盘”。

具体实现：

- 解锁状态作为首屏主视觉。
- 未解锁时突出剩余任务，不让用户误以为系统故障。
- 已有结果以摘要卡展示，减少长段说明。
- 历史记录改为时间线 / 记录流样式。
- 安全提醒继续保持最高层级，风险色不娱乐化。

不改：

- 完整报告解锁判断。
- 历史记录删除能力。
- 安全提示展示条件。

#### C. 树洞频道

目标：从“功能列表页”升级为“校园匿名内容流”。

具体实现：

- 广场页顶部展示温和频道头部和发帖主操作。
- 发帖按钮作为固定或高显著主操作，但不遮挡内容。
- 帖子卡片减少边框感，强化匿名内容、时间、支持反馈。
- 授权拒绝态保持明确，但文案不制造羞辱感。
- 发布页弱化说明卡堆叠，突出文本输入。
- 高风险拦截页继续严肃，不使用装饰性动效。

不改：

- 高风险不公开。
- 预设 reaction toggle。
- 正式账号软删除不暴露后台保留语义。
- 演示账号文案分流。

#### D. 我的频道

目标：从“功能收纳页”升级为“个人与支持中心”。

具体实现：

- 身份卡更简洁，突出昵称、学院、授权状态。
- 帮助资源、报告入口、退出登录使用 WeUI 风格列表单元。
- 演示环境提示只在存在 runtime hints 时展示。
- 退出登录保持危险 / 次危险视觉，不做趣味化。

不改：

- 登录状态管理。
- 授权状态展示。
- 帮助页跳转。

#### E. 问卷作答页组

目标：统一题卡体验，降低模板化表单观感。

具体实现：

- 题目卡片采用更清晰的序号、题干、选项层级。
- 选项行增强点击态和已选态。
- 进度条固定为轻量主线反馈。
- 提交区明确显示未答拦截和加载态。

不改：

- 题库来源。
- 漏答校验。
- 提交接口。
- 评分展示数据。

#### F. 登录、授权、启动、帮助页

目标：只做风格同步，不做深度结构调整。

具体实现：

- 登录页和启动页同步新版背景、按钮和卡片语言。
- 授权页保持信息严肃，不压缩关键说明。
- 帮助页强化热线、校内资源、呼吸建议分组。

不改：

- 登录流程。
- 授权同意 / 拒绝语义。
- 帮助热线行为。

### 2.1.7 管理端处理策略

管理端本阶段不做 Material 化重绘，不引入前端框架。原因是管理后台的首要目标是信息密度、审计清晰和演示稳定。

可做的轻量同步：

- 将 Streamlit CSS 中的品牌色微调为新版 `calm-primary` 系列。
- 保持 KPI、队列、审计日志的高密度布局。
- 不改变后台信息架构和 API 调用。

本阶段如果时间有限，管理端可暂不改，只保证学生端视觉质量。

### 2.1.8 实施顺序

确认方案后，建议按以下顺序执行：

1. 修改全局视觉基座：`app.wxss`、`styles/page.wxss`、`app-shell`。
2. 重绘底部导航：`custom-tab-bar`，必要时同步 `floating-tab-bar`。
3. 重绘基础组件：按钮、卡片、标签、进度条、结果 badge、列表单元。
4. 重绘四个一级频道：首页 / 报告 / 树洞 / 我的。
5. 重绘问卷作答页组和完整报告页。
6. 同步登录、授权、启动、帮助页的基础视觉。
7. 视情况轻量同步管理端 CSS。

### 2.1.9 验收标准

UI 重绘完成后至少满足：

- 不改变核心业务流程和后端契约。
- `测评 / 报告 / 树洞 / 我的` 四个频道视觉上可区分。
- 首页首屏不再表现为普通白卡模板堆叠。
- 高风险、需关注、低风险状态均有文字和颜色双重表达。
- 底部导航不遮挡最后一张卡片或主要按钮。
- 微信开发者工具中主要页面可正常编译、导航和滚动。
- 正式账号 UI 不泄漏后台审计、软删除保留等实现细节。
- 演示账号如需展示实现说明，必须继续通过 `is_demo` 或 runtime hints 分流。

### 2.1.10 明确不做

本阶段不做：

- 不引入 Material UI React 组件库。
- 不迁移到 React / Next.js / Taro / uni-app。
- 不新增评论、私信、联系人、动态头像墙等社交能力。
- 不做复杂拖拽、物理吸附、页面共享元素转场。
- 不做默认深色模式。
- 不用霓虹渐变、泛紫色 SaaS 风、卡通吉祥物化心理主题。

## 3. 新版设计目标

新版学生端必须同时满足以下目标：

1. 从“模块入口页”升级为“主频道导航页”。
2. 视觉气质更接近主流内容型 App，而不是传统表单型系统。
3. 底部导航必须成为主要的一级信息架构入口。
4. 树洞、测评、报告、我的四类核心能力需要在导航结构上被清晰拆开。
5. 高风险内容与安全提醒仍必须保持严肃，不得因整体风格变得娱乐化。
6. 颜色系统必须可继续迭代，不能把当前青绿色过渡方案误当作最终品牌定稿。
7. 最终界面必须仍然让用户感知到“这是一个微信小程序”，而不是完全脱离微信生态语境的原生社交 App 仿制界面。

## 3.1 微信小程序一致性原则

结合 `weui-wxss` 的设计思路，新版学生端必须遵守以下额外原则：

1. 组件结构优先服从微信小程序的原生阅读习惯，再考虑视觉个性化。
2. 页面中的表单、弹窗、状态提示、列表单元等基础控件，应尽可能保持接近微信原生信息密度与对齐关系。
3. 即使底部导航采用更强的产品化视觉，也不能让其破坏微信小程序常见的点击区尺寸、文字可读性和层级秩序。
4. 新版 UI 的创新重点应放在导航组织、内容布局和视觉统一性，而不是强行做大量不符合微信容器习惯的拟原生炫技。

### 3.2 WeUI 参考优先级

结合 `weui-wxss` 的定位，新版学生端应按下面优先级取舍：

1. 一级导航结构与频道组织：允许心语自定义
2. 内容流卡片与频道首页布局：允许心语自定义
3. 基础表单控件：优先参考 WeUI
4. 列表单元、分组、间距密度：优先参考 WeUI
5. 对话框、确认弹层、Toast、状态提示：优先参考 WeUI
6. 高风险提醒与安全支持页：在 WeUI 基础秩序上做心语自己的语义增强

换句话说：

- 导航可以更像 Telegram / 新 iOS；
- 但基础控件不应脱离微信小程序原生体验。

## 4. 导航结构原则

### 4.1 一级导航方向

新版学生端建议采用 4 个一级频道：

1. `树洞`
2. `测评`
3. `报告`
4. `我的`

其中：

- `树洞`：匿名广场、发帖、帖子详情
- `测评`：快速筛查、SDS、SAS、睡眠、UPI
- `报告`：报告摘要、完整报告、历史结果
- `我的`：授权状态、帮助资源、演示模式提示、退出登录

### 4.2 导航形式

新版底部导航不再使用传统“贴底矩形 tabBar”视觉，而应采用“悬浮胶囊式导航栏”：

- 悬浮于页面内容之上
- 与页面底部保留安全边距
- 整体采用大圆角深色半透明容器
- 当前选中项采用滑块式高亮底
- 本阶段仅要求“点击切换”，不要求拖拽滑动选择

### 4.3 页面切换口径

本阶段导航切换只要求支持：

- 点击底部导航切换一级频道
- 当前频道高亮态明显可见
- 页面切换动效平稳
- 当前频道状态可恢复

本阶段不要求：

- 手指拖拽底部高亮块
- 模拟 Telegram 式侧滑物理吸附
- 仿原生 iOS 多段拖动交互

## 5. 视觉方向

### 5.1 整体气质

新版视觉必须从“轻后台式卡片堆叠”转向“主流 App 内容界面”：

- 内容区域更整洁
- 卡片数量减少但单卡完成度提高
- 留白更克制
- 视觉重点从“说明块”转向“主要操作与当前状态”

但必须额外保持：

- 仍然符合微信小程序的阅读节奏；
- 仍然适合单手点击与纵向滚动；
- 仍然保留微信生态常见的克制感，而不是做成完全异质的社交 IM 外壳。

### 5.2 底部导航栏风格

底部导航栏建议采用如下视觉语言：

- 背景：深色半透明或拟磨砂深灰容器
- 形状：超大圆角胶囊
- 高度：显著高于传统 tabBar
- 图标：线性或简化面性图标，保持统一风格
- 文字：短标签，避免长句
- 选中态：单独的高亮色块或亮色胶囊底
- 未选中态：中性低对比文字与图标

补充约束：

- 虽然视觉参考 Telegram / 新 iOS，但图标密度、文字尺寸和点击区域仍应回到微信小程序可接受范围；
- 底部导航不应因为追求视觉像素级模仿，而牺牲微信小程序中的触达效率与页面清晰度。

### 5.3 选中态表达

当前选中项的高亮必须满足：

- 一眼可区分当前频道
- 高亮底块可在点击切换时平滑移动
- 图标与文字色同时切换
- 高亮色不使用攻击性过强的霓虹色

推荐高亮方向：

- 蓝绿色或冷青色高亮底
- 保留品牌体系中的 calm / trusted 感受
- 不直接复用高风险红或暖色强调

当前说明：

- 这里仍然保留“蓝绿色 / 冷青色”作为暂定高亮方向，仅仅是为了与现有项目视觉遗产保持最低限度连续性；
- 若后续找到更成熟的新品牌主色，高亮底块颜色应允许整体替换，不应与当前青绿色体系强绑定。

## 6. 颜色系统

### 6.1 当前阶段的颜色策略

当前阶段的配色目标，不是立即确定最终品牌色，而是：

1. 先保留现有青绿色方向，降低重构阶段的变量数量；
2. 先完成导航结构与组件层级升级；
3. 再在后续迭代中，把当前青绿色体系替换成更成熟的新版配色。

也就是说：

- 青绿色当前可以继续使用；
- 但它不是最终答案；
- 文档中的所有颜色都必须以“过渡 token”而不是“最终品牌定稿”来理解。

2. 不允许在业务组件里继续散落写死 `#FFFFFF`、`#123F3A`、`#2D7C84` 之类的值；
3. 底部悬浮导航栏、卡片背景、输入框、风险提示卡、按钮主次态都必须有双主题映射；
4. 高风险页与安全提示页在暗色主题下仍必须保持严肃可读，不能为了“夜间模式”做成酷炫危险风格。
5. 主题切换后的基础控件层级仍应尽量兼容 `weui-wxss` 所代表的微信原生视觉秩序。

补充实现口径：

- 参考 `weui-wxss` 的暗色模式思路，未来若正式落地，可在页面根节点增加统一主题标记，例如：
  - `data-weui-theme="light"`
  - `data-weui-theme="dark"`
- 即使项目最终不直接引入 `weui-wxss` 样式文件，也建议保留这一类“根容器主题标记”的实现思路，便于统一主题切换和组件映射。

## 8. 布局规则

### 8.1 页面结构

新版页面建议采用：

- 顶部状态标题区
- 中部主内容流
- 底部悬浮导航栏

并额外要求：

- 明亮 / 暗色两种主题下都必须保证同一信息架构成立；
- 不能因为切到暗色主题，就让层级结构依赖颜色本身去区分；
- 结构层次必须先由 spacing、surface、border 和 typography 支撑，再由颜色增强。

### 8.2 页面安全区

必须重点处理底部导航的安全区：

- 导航栏底部不得直接贴边
- iPhone 底部 Home Indicator 上方必须保留安全边距
- 页面底部内容要为悬浮导航栏预留滚动空间

### 8.3 滚动区域

为避免悬浮导航遮挡正文：

- 主内容列表底部需增加额外 padding
- 最后一张卡片不能被悬浮导航压住
- 删除确认、风险提示等弹层需高于悬浮导航层级

## 9. 组件策略

### 9.1 学生端新增核心组件

在现有组件之外，新版应优先新增：

- `floating-tab-bar`
- `tab-pill`
- `channel-header`
- `content-section-card`
- `quick-action-row`

### 9.2 旧组件处理策略

现有组件不必一次性废弃，但需要逐步重构：

- `hero-card`：保留语义，但弱化“说明型大卡片”的占比
- `treehole-post-card`：改得更接近内容流卡片
- `primary-button / secondary-button / danger-button`：统一到新版按钮系统
- `confirm-modal`：保留，但视觉需更简洁

### 9.3 与 WeUI 的关系

`weui-wxss` 不应被理解为“必须整库直接引入替换现有样式”，而应被理解为一套微信官方的基础界面参考标准。对本项目的正确使用方式是：

1. 参考其基础控件的对齐、留白、字号和层级关系；
2. 参考其明暗主题切换口径，而不是机械照搬其全部视觉；
3. 在新版组件中吸收其“微信原生一致性”原则，同时保留心语自己的导航结构与品牌表达。

若未来确需落地实现，可优先采用以下方式之一：

- 方式 A：继续保持当前自有组件体系，仅借鉴 `weui-wxss` 的结构与 token 组织方式；
- 方式 B：对表单、列表单元、弹窗、按钮等基础层组件进行 WeUI 风格校准，而把底部悬浮导航和频道布局保持为心语自定义风格。

### 9.4 建议对齐 WeUI 的基础能力

未来若正式进入新版重构，以下基础能力建议优先对齐 `weui-wxss` 的结构规律：

- 按钮：主按钮、次按钮、危险按钮、禁用态
- 输入区：输入框、文本域、输入提示、错误态
- 列表单元：频道设置项、我的页面入口项、状态项
- 对话框：删除确认、授权提醒、退出登录确认
- 状态提示：Toast、成功提示、错误提示、轻量提醒
- 进度表现：问卷进度、报告解锁进度、提交反馈

这些区域的目标不是“长得和 WeUI 一模一样”，而是：

- 结构秩序更像微信官方小程序；
- 学习成本更低；
- 与底部导航的强自定义形成平衡。

## 10. 底部导航实现办法

### 10.1 本阶段实现目标

本阶段底部导航的实现范围如下：

- 自定义悬浮底部导航栏
- 点击切换一级频道
- 当前选中项高亮底块平移动画
- 图标与文字联动切换

同时要求：

- 明亮与暗色主题下都必须有完整视觉映射；
- 当前选中态的滑块式高亮底在两种主题下都要明显，但不能刺眼；
- 主题切换不应改变导航结构，只改变视觉 token。

### 10.2 推荐实现方式

微信小程序中建议采用：

1. 不使用原生默认 `tabBar` 视觉
2. 页面主体使用自定义导航容器
3. 当前选中项以单独 `active-pill` 元素实现
4. 点击某个频道时更新 `activeIndex`
5. 通过 `transform: translateX(...)` 做高亮块平移动画

补充建议：

6. 页面基础区块、列表单元和表单区可参考 `weui-wxss` 的结构密度与分组方式；
7. 底部导航是允许自定义创新的部分，但按钮、输入、提示等基础能力应尽量回归微信原生语境。

### 10.3 不建议的实现方式

本阶段不建议：

- 上来就做手势拖拽选中
- 为了仿 iOS 而堆叠过重动画
- 在小程序里硬做复杂物理滑块吸附
- 先做高保真视觉再回头补真实信息架构
- 让所有基础组件都脱离微信原生习惯，只剩底部导航像某个外部 App

## 11.1 微信原生与品牌自定义的边界

未来真正落地时，应明确区分：

### 必须更像微信原生的区域

- 表单输入
- 列表单元
- 系统确认弹窗
- Toast 与轻量状态反馈
- 我的页设置分组

### 可以更像 Telegram / 新 iOS 的区域

- 底部悬浮导航栏
- 一级频道首页布局
- 树洞内容流视觉
- 报告频道摘要卡
- 顶部频道头部与快捷操作区

这个边界是新版 UI 成败的关键：

- 如果所有地方都像 Telegram，就会失去微信小程序的一致性；
- 如果所有地方都像 WeUI，就无法完成“更现代的频道化升级”。

## 13. 页面级落地清单

本节用于回答“未来真的开始前端重构时，每个频道该落哪些页面、每个页面具体改什么、哪些先改、哪些后改”。

页面级落地清单的原则是：

1. 先改信息架构，再改单页装饰；
2. 先改一级频道首页，再改二级详情页；
3. 先改用户每天高频触达页面，再改低频页面；
4. 只改 UI / UX，不主动改后端契约，除非页面结构明确无法承载现有接口。

## 14. 文案规则

新版文案必须遵守：

1. 学生端只说用户视角，不泄漏后台实现细节。
2. 演示说明如必须存在，应仅在演示账号或明确演示模式下显示。
3. 不把删除、审核、风险提示写成系统内幕说明。
4. 不使用诊断式、确定性、命令式措辞。

## 15. 重构顺序建议

若未来真正启动新版前端重构，建议顺序如下：

1. 先定义明亮 / 暗色双主题 token 与导航色板
2. 再重构一级导航结构与底部悬浮导航栏
3. 再拆分首页为 `树洞 / 测评 / 报告 / 我的`
4. 再统一新版设计令牌与公共组件
5. 最后再逐页优化空状态、加载态、过渡动画

落地执行时，页面优先级建议如下：

1. 树洞广场页
2. 树洞发布页
3. 测评频道首页
4. 报告频道首页
5. 我的频道首页
6. 问卷作答页组
7. 帖子详情页
8. 完整报告详情页
9. 帮助资源页

## 16. 与现有版本的关系

本文件当前不是替代现有 `FRONTEND_GUIDELINES.md` 的立即执行规范，而是面向下一阶段 UI / UX 重构的设计蓝图。若未来正式采用该方向，应在启动新版前端重构前，明确：

- 是否保留当前信息架构；
- 是否正式采用底部悬浮导航；
- 是否正式支持明亮 / 暗色双主题；
- 是否继续沿用当前青绿色过渡方案，还是整体替换为新的品牌色；
- 是否只把 `weui-wxss` 作为参考标准，还是在基础控件层面局部对齐其实现方式；
- 哪些页面只做视觉迁移，哪些页面要做频道重构；
- 哪些交互仍保持当前业务状态机不变。
