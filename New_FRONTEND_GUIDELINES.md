# 心语新版前端指南

## 1. 文档目的

本文档用于定义心语学生端下一阶段的新版 UI / UX 方向。它不是对现有 `FRONTEND_GUIDELINES.md` 的直接覆盖，而是面向未来重构阶段的候选设计规范，重点回答：

- 如何把当前“单页向下滚动的入口式结构”升级为更接近主流 App 的导航结构；
- 如何在不引入即时通讯业务模型的前提下，借鉴 Telegram 与新一代 iOS 的底部悬浮导航语言；
- 如何在吸收微信官方小程序基础样式思想的前提下，避免把 UI 做成“脱离微信容器语境的外来 App 壳”；
- 如何在保持“平静、可信、低压”的心理健康产品气质下，完成更现代、更高完成度的学生端体验。

本文件当前主要服务于学生端小程序的新版视觉与交互设计；管理后台仍以现有 `FRONTEND_GUIDELINES.md` 为准。

参考来源：

- 当前新版 UI 设计除了参考 Telegram 风格的底部悬浮导航语言外，也明确参考微信官方设计团队开源的小程序基础样式库 `weui-wxss`：
  - https://github.com/Tencent/weui-wxss/
- `weui-wxss` 本身提供的是“与微信原生视觉体验一致的基础样式库”，核心价值在于：按钮、表单、列表单元、对话框、进度、提示、文章内容等基础界面元素都遵循微信用户已熟悉的阅读与操作节奏；
- 后续所有学生端新版视觉与交互设计，应理解为“Telegram / 新 iOS 导航语言 + 微信小程序原生一致性约束”的结合方案，而不是单纯照搬外部 App 的视觉外观。

特别说明：

- 本文件中当前仍保留的青绿色基调，只是因为现阶段尚未形成更成熟、更具辨识度的新品牌配色方案，因此暂时沿用项目既有的视觉记忆；
- 这套青绿色当前应被视为“过渡色系”，用于支撑新版导航结构、信息架构与交互层级的重构；
- 后续一旦形成更完整的新版主色系统，本文件中的品牌主色、导航高亮色与双主题色板都应继续修改，而不应被视为最终定稿。

## 2. 适用范围

- 学生端：微信小程序新版 UI / UX 重构
- 管理端：本文件不作为直接约束

## 3. 新版设计目标

新版学生端必须同时满足以下目标：

1. 从“模块入口页”升级为“主频道导航页”。
2. 视觉气质更接近主流内容型 App，而不是传统表单型系统。
3. 底部导航必须成为主要的一级信息架构入口。
4. 树洞、测评、报告、我的四类核心能力需要在导航结构上被清晰拆开。
5. 高风险内容与安全提醒仍必须保持严肃，不得因整体风格变得娱乐化。
6. 颜色系统必须可继续迭代，不能把当前青绿色过渡方案误当作最终品牌定稿。
7. 最终界面必须仍然让用户感知到“这是一个微信小程序”，而不是完全脱离微信生态语境的原生社交 App 仿制界面。

## 3.1 微信小程序一致性原则

结合 `weui-wxss` 的设计思路，新版学生端必须遵守以下额外原则：

1. 组件结构优先服从微信小程序的原生阅读习惯，再考虑视觉个性化。
2. 页面中的表单、弹窗、状态提示、列表单元等基础控件，应尽可能保持接近微信原生信息密度与对齐关系。
3. 即使底部导航采用更强的产品化视觉，也不能让其破坏微信小程序常见的点击区尺寸、文字可读性和层级秩序。
4. 新版 UI 的创新重点应放在导航组织、内容布局和视觉统一性，而不是强行做大量不符合微信容器习惯的拟原生炫技。

### 3.2 WeUI 参考优先级

结合 `weui-wxss` 的定位，新版学生端应按下面优先级取舍：

1. 一级导航结构与频道组织：允许心语自定义
2. 内容流卡片与频道首页布局：允许心语自定义
3. 基础表单控件：优先参考 WeUI
4. 列表单元、分组、间距密度：优先参考 WeUI
5. 对话框、确认弹层、Toast、状态提示：优先参考 WeUI
6. 高风险提醒与安全支持页：在 WeUI 基础秩序上做心语自己的语义增强

换句话说：

- 导航可以更像 Telegram / 新 iOS；
- 但基础控件不应脱离微信小程序原生体验。

## 4. 导航结构原则

### 4.1 一级导航方向

新版学生端建议采用 4 个一级频道：

1. `树洞`
2. `测评`
3. `报告`
4. `我的`

其中：

- `树洞`：匿名广场、发帖、帖子详情
- `测评`：快速筛查、SDS、SAS、睡眠、UPI
- `报告`：报告摘要、完整报告、历史结果
- `我的`：授权状态、帮助资源、演示模式提示、退出登录

### 4.2 导航形式

新版底部导航不再使用传统“贴底矩形 tabBar”视觉，而应采用“悬浮胶囊式导航栏”：

- 悬浮于页面内容之上
- 与页面底部保留安全边距
- 整体采用大圆角深色半透明容器
- 当前选中项采用滑块式高亮底
- 本阶段仅要求“点击切换”，不要求拖拽滑动选择

### 4.3 页面切换口径

本阶段导航切换只要求支持：

- 点击底部导航切换一级频道
- 当前频道高亮态明显可见
- 页面切换动效平稳
- 当前频道状态可恢复

本阶段不要求：

- 手指拖拽底部高亮块
- 模拟 Telegram 式侧滑物理吸附
- 仿原生 iOS 多段拖动交互

## 5. 视觉方向

### 5.1 整体气质

新版视觉必须从“轻后台式卡片堆叠”转向“主流 App 内容界面”：

- 内容区域更整洁
- 卡片数量减少但单卡完成度提高
- 留白更克制
- 视觉重点从“说明块”转向“主要操作与当前状态”

但必须额外保持：

- 仍然符合微信小程序的阅读节奏；
- 仍然适合单手点击与纵向滚动；
- 仍然保留微信生态常见的克制感，而不是做成完全异质的社交 IM 外壳。

### 5.2 底部导航栏风格

底部导航栏建议采用如下视觉语言：

- 背景：深色半透明或拟磨砂深灰容器
- 形状：超大圆角胶囊
- 高度：显著高于传统 tabBar
- 图标：线性或简化面性图标，保持统一风格
- 文字：短标签，避免长句
- 选中态：单独的高亮色块或亮色胶囊底
- 未选中态：中性低对比文字与图标

补充约束：

- 虽然视觉参考 Telegram / 新 iOS，但图标密度、文字尺寸和点击区域仍应回到微信小程序可接受范围；
- 底部导航不应因为追求视觉像素级模仿，而牺牲微信小程序中的触达效率与页面清晰度。

### 5.3 选中态表达

当前选中项的高亮必须满足：

- 一眼可区分当前频道
- 高亮底块可在点击切换时平滑移动
- 图标与文字色同时切换
- 高亮色不使用攻击性过强的霓虹色

推荐高亮方向：

- 蓝绿色或冷青色高亮底
- 保留品牌体系中的 calm / trusted 感受
- 不直接复用高风险红或暖色强调

当前说明：

- 这里仍然保留“蓝绿色 / 冷青色”作为暂定高亮方向，仅仅是为了与现有项目视觉遗产保持最低限度连续性；
- 若后续找到更成熟的新品牌主色，高亮底块颜色应允许整体替换，不应与当前青绿色体系强绑定。

## 6. 颜色系统

### 6.1 当前阶段的颜色策略

当前阶段的配色目标，不是立即确定最终品牌色，而是：

1. 先保留现有青绿色方向，降低重构阶段的变量数量；
2. 先完成导航结构与组件层级升级；
3. 再在后续迭代中，把当前青绿色体系替换成更成熟的新版配色。

也就是说：

- 青绿色当前可以继续使用；
- 但它不是最终答案；
- 文档中的所有颜色都必须以“过渡 token”而不是“最终品牌定稿”来理解。

### 6.2 新版品牌主色（过渡方案）

当前过渡方案继续保留“心语”原有青绿色基因，但提升现代度：

| `brand-bg-top` | `#F3F8F7` | 页面顶部大背景 |
| `brand-bg-bottom` | `#EDF4F2` | 页面底部背景过渡 |
| `brand-primary` | `#2D7C84` | 主品牌色 |
| `brand-primary-strong` | `#185B62` | 标题 / 深强调 |
| `brand-accent` | `#4F9EBD` | 导航高亮 / 信息强调 |

注意：

- 上述主色当前只是过渡方案；
- `brand-primary` 与 `brand-accent` 后续允许整体替换；
- 新版重构时必须优先让组件引用语义 token，而不是直接在组件里写死这些值。

### 6.5 暗色主题设计方向

暗色主题的目标不是“纯黑模式”，而是“低刺激夜间模式”：

- 背景偏深海青灰，而不是纯黑
- 卡片层级清楚，但不刺眼
- 文本对比充足，但不做高亮白光
- 导航栏仍保留深色胶囊壳体

### 6.6 主题切换规则

未来若正式支持明亮 / 暗色双主题，必须遵守：

1. 所有页面与组件都通过主题 token 取色；
2. 不允许在业务组件里继续散落写死 `#FFFFFF`、`#123F3A`、`#2D7C84` 之类的值；
3. 底部悬浮导航栏、卡片背景、输入框、风险提示卡、按钮主次态都必须有双主题映射；
4. 高风险页与安全提示页在暗色主题下仍必须保持严肃可读，不能为了“夜间模式”做成酷炫危险风格。
5. 主题切换后的基础控件层级仍应尽量兼容 `weui-wxss` 所代表的微信原生视觉秩序。

补充实现口径：

- 参考 `weui-wxss` 的暗色模式思路，未来若正式落地，可在页面根节点增加统一主题标记，例如：
  - `data-weui-theme="light"`
  - `data-weui-theme="dark"`
- 即使项目最终不直接引入 `weui-wxss` 样式文件，也建议保留这一类“根容器主题标记”的实现思路，便于统一主题切换和组件映射。

### 6.7 语义色

风险相关语义色仍保持克制：

| `danger-700` | `#A92A2A` | 严肃主操作 |

说明：

- 语义色当前可以在明亮 / 暗色主题间基本保持一致；
- 后续若新版品牌色整体替换，语义色可以微调，但不得牺牲风险状态识别度。

## 7. 字体系统

### 7.1 字体栈

新版学生端继续使用系统友好中文字体：

- `PingFang SC`
- `Hiragino Sans GB`
- `Microsoft YaHei`
- `sans-serif`

## 8. 布局规则

### 8.1 页面结构

新版页面建议采用：

- 顶部状态标题区
- 中部主内容流
- 底部悬浮导航栏

并额外要求：

- 明亮 / 暗色两种主题下都必须保证同一信息架构成立；
- 不能因为切到暗色主题，就让层级结构依赖颜色本身去区分；
- 结构层次必须先由 spacing、surface、border 和 typography 支撑，再由颜色增强。

### 8.2 页面安全区

必须重点处理底部导航的安全区：

- 导航栏底部不得直接贴边
- iPhone 底部 Home Indicator 上方必须保留安全边距
- 页面底部内容要为悬浮导航栏预留滚动空间

### 8.3 滚动区域

为避免悬浮导航遮挡正文：

- 主内容列表底部需增加额外 padding
- 最后一张卡片不能被悬浮导航压住
- 删除确认、风险提示等弹层需高于悬浮导航层级

## 9. 组件策略

### 9.1 学生端新增核心组件

在现有组件之外，新版应优先新增：

- `floating-tab-bar`
- `tab-pill`
- `channel-header`
- `content-section-card`
- `quick-action-row`

### 9.2 旧组件处理策略

现有组件不必一次性废弃，但需要逐步重构：

- `hero-card`：保留语义，但弱化“说明型大卡片”的占比
- `treehole-post-card`：改得更接近内容流卡片
- `primary-button / secondary-button / danger-button`：统一到新版按钮系统
- `confirm-modal`：保留，但视觉需更简洁

### 9.3 与 WeUI 的关系

`weui-wxss` 不应被理解为“必须整库直接引入替换现有样式”，而应被理解为一套微信官方的基础界面参考标准。对本项目的正确使用方式是：

1. 参考其基础控件的对齐、留白、字号和层级关系；
2. 参考其明暗主题切换口径，而不是机械照搬其全部视觉；
3. 在新版组件中吸收其“微信原生一致性”原则，同时保留心语自己的导航结构与品牌表达。

若未来确需落地实现，可优先采用以下方式之一：

- 方式 A：继续保持当前自有组件体系，仅借鉴 `weui-wxss` 的结构与 token 组织方式；
- 方式 B：对表单、列表单元、弹窗、按钮等基础层组件进行 WeUI 风格校准，而把底部悬浮导航和频道布局保持为心语自定义风格。

### 9.4 建议对齐 WeUI 的基础能力

未来若正式进入新版重构，以下基础能力建议优先对齐 `weui-wxss` 的结构规律：

- 按钮：主按钮、次按钮、危险按钮、禁用态
- 输入区：输入框、文本域、输入提示、错误态
- 列表单元：频道设置项、我的页面入口项、状态项
- 对话框：删除确认、授权提醒、退出登录确认
- 状态提示：Toast、成功提示、错误提示、轻量提醒
- 进度表现：问卷进度、报告解锁进度、提交反馈

这些区域的目标不是“长得和 WeUI 一模一样”，而是：

- 结构秩序更像微信官方小程序；
- 学习成本更低；
- 与底部导航的强自定义形成平衡。

## 10. 底部导航实现办法

### 10.1 本阶段实现目标

本阶段底部导航的实现范围如下：

- 自定义悬浮底部导航栏
- 点击切换一级频道
- 当前选中项高亮底块平移动画
- 图标与文字联动切换

同时要求：

- 明亮与暗色主题下都必须有完整视觉映射；
- 当前选中态的滑块式高亮底在两种主题下都要明显，但不能刺眼；
- 主题切换不应改变导航结构，只改变视觉 token。

### 10.2 推荐实现方式

微信小程序中建议采用：

1. 不使用原生默认 `tabBar` 视觉
2. 页面主体使用自定义导航容器
3. 当前选中项以单独 `active-pill` 元素实现
4. 点击某个频道时更新 `activeIndex`
5. 通过 `transform: translateX(...)` 做高亮块平移动画

补充建议：

6. 页面基础区块、列表单元和表单区可参考 `weui-wxss` 的结构密度与分组方式；
7. 底部导航是允许自定义创新的部分，但按钮、输入、提示等基础能力应尽量回归微信原生语境。

### 10.3 不建议的实现方式

本阶段不建议：

- 上来就做手势拖拽选中
- 为了仿 iOS 而堆叠过重动画
- 在小程序里硬做复杂物理滑块吸附
- 先做高保真视觉再回头补真实信息架构
- 让所有基础组件都脱离微信原生习惯，只剩底部导航像某个外部 App

## 11.1 微信原生与品牌自定义的边界

未来真正落地时，应明确区分：

### 必须更像微信原生的区域

- 表单输入
- 列表单元
- 系统确认弹窗
- Toast 与轻量状态反馈
- 我的页设置分组

### 可以更像 Telegram / 新 iOS 的区域

- 底部悬浮导航栏
- 一级频道首页布局
- 树洞内容流视觉
- 报告频道摘要卡
- 顶部频道头部与快捷操作区

这个边界是新版 UI 成败的关键：

- 如果所有地方都像 Telegram，就会失去微信小程序的一致性；
- 如果所有地方都像 WeUI，就无法完成“更现代的频道化升级”。

## 12. 频道页面建议

### 12.1 树洞频道

- 强调内容流
- 发帖按钮固定为显著主操作
- 公开内容卡片更像“消息流”而不是“说明卡”

### 12.2 测评频道

- 不再把所有问卷塞入首页长页
- 采用清晰的任务型入口
- 强化必做 / 可选层级

### 12.3 报告频道

- 强化当前状态：未完成 / 已解锁 / 已有结果
- 报告摘要卡优先于大段说明

### 12.4 我的频道

- 集中放置授权状态、帮助资源、演示信息、退出登录
- 避免再与主功能入口混合

## 13. 页面级落地清单

本节用于回答“未来真的开始前端重构时，每个频道该落哪些页面、每个页面具体改什么、哪些先改、哪些后改”。

页面级落地清单的原则是：

1. 先改信息架构，再改单页装饰；
2. 先改一级频道首页，再改二级详情页；
3. 先改用户每天高频触达页面，再改低频页面；
4. 只改 UI / UX，不主动改后端契约，除非页面结构明确无法承载现有接口。

### 13.1 树洞频道页面清单

树洞频道是新版视觉优先级最高的频道，应最先完成。

#### A. 树洞广场页

现有页面：

- `miniprogram/pages/treehole/feed/index`

新版定位：

- 树洞频道一级首页
- 承担内容流浏览、快速进入发帖、进入帖子详情三项职责

必须重构的内容：

- 从“功能页”改成“内容流首页”
- 顶部加入频道标题与轻量状态摘要
- 内容卡片改成更像消息流/帖子流
- 发帖按钮改成显著主操作
- 底部悬浮导航常驻

应保留的业务逻辑：

- 广场只显示允许公开的帖子
- 预设互动逻辑不变
- 删除后的帖子不再出现在学生端广场

暂时不做：

- 评论
- 私信
- 联系人关系

#### B. 树洞发布页

现有页面：

- `miniprogram/pages/treehole/create/index`

新版定位：

- 从树洞频道内进入的二级“创作页”

必须重构的内容：

- 弱化“说明卡堆叠”，强化输入主任务
- 规则说明缩成更紧凑的提示区
- 字数区、输入区、发布按钮形成单一创作流
- 成功 / 拦截分流后的视觉反馈更清晰

应保留的业务逻辑：

- 当前文本 AI 风险决定是否拦截
- 高风险仍进安全支持流程
- 问卷/历史高风险只做后台 follow-up

暂时不做：

- 草稿系统
- 多媒体上传

#### C. 帖子详情页

现有页面：

- `miniprogram/pages/treehole/detail/index`

新版定位：

- 树洞频道内的详情页

必须重构的内容：

- 详情头部更简洁
- 支持反馈与帖子正文层级更清楚
- 删除相关提示文案继续按 `is_demo` 分流
- 正式账号不暴露后台保留语义

应保留的业务逻辑：

- 只有本人公开帖子可删
- 删除后进入学生端移除状态
- 演示账号允许保留实现解释文案

### 13.2 测评频道页面清单

测评频道的目标是从“首页模块入口集合”改为“任务型频道”。

#### A. 测评频道首页

现有页面来源：

- 目前主要逻辑散落在 `miniprogram/pages/home/index`

新版定位：

- 测评频道一级首页

必须重构的内容：

- 独立承载五份问卷入口
- 必做 / 可选区分明确
- 当前进度、剩余任务、完整报告解锁状态清晰可见
- 入口卡片按任务优先级组织，不再只是静态模块列表

应保留的业务逻辑：

- 必做 70 题链路
- UPI 不阻塞完整报告

#### B. 问卷作答页组

现有页面：

- `miniprogram/pages/questionnaires/screen/index`
- `miniprogram/pages/questionnaires/sds/index`
- `miniprogram/pages/questionnaires/sas/index`
- `miniprogram/pages/questionnaires/sleep/index`
- `miniprogram/pages/questionnaires/upi/index`

新版定位：

- 统一问卷作答体验的一组同构页面

必须重构的内容：

- 统一题卡样式
- 统一按钮区、分页区、提交流程反馈
- 明暗主题下保证可读性
- 降低“表单系统感”，增强“任务流程感”

应保留的业务逻辑：

- 现有问卷提交接口
- 现有评分与解锁规则
- 未答题阻止提交

暂时不做：

- 动态题库编辑
- 实时保存草稿到后端

### 13.3 报告频道页面清单

报告频道的目标是让用户一进入就理解“我当前有没有解锁、我已经完成到哪一步、我现在能看什么”。

#### A. 报告频道首页

现有页面：

- `miniprogram/pages/reports/index`

新版定位：

- 报告频道一级首页

必须重构的内容：

- 把“锁定 / 部分完成 / 已解锁”做成强状态页
- 摘要卡比说明文字更优先
- 让“下一步去做哪份问卷”更明确

应保留的业务逻辑：

- 70 题未完成时不可进完整报告
- 已有量表结果可局部展示

#### B. 完整报告详情页

现有页面：

- `miniprogram/pages/reports/full/index`

新版定位：

- 报告频道二级详情页

必须重构的内容：

- 降低静态长文观感
- 提升结构化摘要、风险标签、建议区层级
- 保留免责声明但不要压过主要结果

应保留的业务逻辑：

- 只有达到解锁条件时可进入
- 不输出诊断式结论

### 13.4 我的频道页面清单

我的频道不应再是“首页剩余功能杂糅区”，而是一个清晰的个人与辅助设置频道。

#### A. 我的频道首页

新版定位：

- 我的频道一级首页

建议承载内容：

- 当前学生身份摘要
- 授权状态
- 帮助资源入口
- 演示模式提示（如存在）
- 退出登录

必须重构的内容：

- 列表单元尽量向 WeUI 对齐
- 让“设置 / 辅助 / 帮助”比“功能入口”更像个人页面

#### B. 帮助资源页

现有业务对应：

- `S15 帮助页`

新版定位：

- 我的频道下的独立帮助页或快捷入口页

必须重构的内容：

- 热线、校内资源、呼吸调节建议做分组
- 高风险帮助信息层级高于普通自助建议

### 13.5 当前阶段不优先重构的页面

以下页面在未来真正进入重构时，可以后置：

- 登录页的高级视觉打磨
- 授权页的深度动画
- 极端空状态页的装饰效果
- 管理后台对应页面

原因：

- 这些页面当前不是新版导航体验的核心；
- 它们不决定新版一级信息架构是否成立；
- 应优先保证频道化结构先跑通。

## 14. 文案规则

新版文案必须遵守：

1. 学生端只说用户视角，不泄漏后台实现细节。
2. 演示说明如必须存在，应仅在演示账号或明确演示模式下显示。
3. 不把删除、审核、风险提示写成系统内幕说明。
4. 不使用诊断式、确定性、命令式措辞。

## 15. 重构顺序建议

若未来真正启动新版前端重构，建议顺序如下：

1. 先定义明亮 / 暗色双主题 token 与导航色板
2. 再重构一级导航结构与底部悬浮导航栏
3. 再拆分首页为 `树洞 / 测评 / 报告 / 我的`
4. 再统一新版设计令牌与公共组件
5. 最后再逐页优化空状态、加载态、过渡动画

落地执行时，页面优先级建议如下：

1. 树洞广场页
2. 树洞发布页
3. 测评频道首页
4. 报告频道首页
5. 我的频道首页
6. 问卷作答页组
7. 帖子详情页
8. 完整报告详情页
9. 帮助资源页

## 16. 与现有版本的关系

本文件当前不是替代现有 `FRONTEND_GUIDELINES.md` 的立即执行规范，而是面向下一阶段 UI / UX 重构的设计蓝图。若未来正式采用该方向，应在启动新版前端重构前，明确：

- 是否保留当前信息架构；
- 是否正式采用底部悬浮导航；
- 是否正式支持明亮 / 暗色双主题；
- 是否继续沿用当前青绿色过渡方案，还是整体替换为新的品牌色；
- 是否只把 `weui-wxss` 作为参考标准，还是在基础控件层面局部对齐其实现方式；
- 哪些页面只做视觉迁移，哪些页面要做频道重构；
- 哪些交互仍保持当前业务状态机不变。
